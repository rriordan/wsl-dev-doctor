# v0.1.0 — Initial MVP

First public release of WSL Dev Environment Doctor: a safe WSL/Linux developer-environment diagnostic CLI with a controlled, opt-in updater planner.

## What shipped

### Diagnostics

- Read-only checks for WSL detection, stale `PATH` entries, root disk capacity, Docker daemon availability, listening TCP ports, and NVIDIA GPU visibility.
- Comprehensive inventory of 60 developer binaries across system packages, Python, JavaScript, source control, AI coding, local LLM, containers, cloud, build, and editor categories.
- `present` / `not_present` inventory status; absence is data, not a health warning.
- Best-effort version capture for detected binaries.
- Markdown and JSON reports with prioritized remediation.
- Environment-variable inventory records names only; values are never collected or emitted.
- `--fail-on warn|fail` for shell or CI automation.

### Controlled updater

- Dry-run update planning by default.
- Dedicated updater support for detected Claude, Codex, and uv installations.
- Optional global JavaScript package updates for npm, pnpm, and Yarn.
- Explicit package-manager gates:
  - `--include-system-packages` for APT upgrades.
  - `--include-global-js-packages` for global JavaScript packages.
  - `--include-homebrew` for Homebrew refreshes.
- Execution requires both `--apply` and `--yes`.
- Terminal interactive selector with arrow-key navigation and spacebar selection.
- Built-in presets plus validated custom TOML presets at `~/.config/wsl-dev-doctor/presets.toml`.
- Structured argv execution only: custom presets select known tool IDs and cannot inject arbitrary shell commands.

## Validation

- Local: Ruff formatting/lint, MyPy, and pytest (**17 tests**) pass.
- Hosted CI: passing on release commit `da84723` before this documentation-only release preparation commit.
- CI exercises formatting, linting, type checking, tests, and a real JSON CLI report.

## Known limitations

- Linux-side diagnostics only; Windows-host interrogation is not included yet.
- Docker and GPU results depend on their respective CLIs being accessible.
- Update source detection is not yet installation-method-aware; when provenance is ambiguous, future versions should prefer `unsupported` over guessing.
- The updater does not execute unless explicitly requested with `--apply --yes`.
